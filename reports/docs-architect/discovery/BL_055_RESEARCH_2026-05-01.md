# BL-055 Research — direct bot-to-portal ticket exchange

Replaces the bot → mini_app → `/api/auth/exchange-miniapp-to-portal`
indirection in the payout flow with a direct bot → API → portal hop.

Mode: **research / planning** per CLAUDE.md "Phase mode discipline" —
critical-eye lens on the plan in `tmp/PROMPT_BL_055_autonomous (1).md`.

## A. Existing `/api/auth/exchange-miniapp-to-portal`

| What | Where |
|---|---|
| Handler | `src/api/routers/auth.py:219` `exchange_miniapp_to_portal` |
| Auth | `Depends(get_current_user_from_mini_app)` — requires mini_app JWT bearer |
| Response | `TicketResponse` (`src/api/schemas/auth.py:13`) — `{ticket, portal_url, expires_in}` |
| Ticket payload | `{sub: str(user.id), tg, plan, jti, aud="web_portal", exp, iat}` |
| Ticket signing | `pyjwt.encode(payload, settings.jwt_secret, algorithm=settings.jwt_algorithm)` |
| Redis bookkeeping | `setex("auth:ticket:jti:{jti}", ttl, json{user_id, issued_at, ip})` — one-shot |
| TTL | `settings.ticket_jwt_ttl_seconds` (default 300s) |
| Counterpart consume | `POST /api/auth/consume-ticket` (`auth.py:292`) — decodes ticket with `audience="web_portal"`, deletes jti, mints `aud="web_portal"` access JWT |
| Portal landing | `${web_portal_url}/login/ticket?ticket=<jwt>&redirect=<encoded>` (FE-side concat in `useOpenInWebPortal.ts:22`) |

Implication for BL-055: the new endpoint must issue a ticket whose
**payload shape is identical** to the existing one, so the existing
`consume-ticket` handler accepts it without modification. The redirect
target is a URL query parameter (not a JWT claim) — the portal's
`safeRedirect` validator parses `?redirect=` independently of the JWT.

## Возражения и риски (per CLAUDE.md "Research reports — Objections section")

### O.1 — A4 ticket payload mismatch with consume-ticket (high)

**Issue.** Plan A4 specifies ticket claims as
`{"sub": telegram_id, "iat": now, "exp": now + 300, "purpose": "portal-login", "redirect": redirect_path}`.
The existing `/api/auth/consume-ticket` handler (`auth.py:315-326`)
decodes the ticket with `audience="web_portal"` and reads
`payload["sub"]` (cast to `int(payload["sub"])`),
`payload["plan"]`, `payload["tg"]`, plus the Redis `jti` lookup.

A ticket issued per A4-as-written would fail at `audience="web_portal"`
(no `aud` claim), and even if that's fixed, `sub` would carry
`telegram_id` not `user.id` — corrupting downstream
`create_jwt_token(user_id=user_id, ...)` in line 379.

**Recommendation (Type 2 ADAPT).** Mirror the existing exchange-miniapp
ticket payload: `{sub: str(user.id), tg, plan, jti, aud: "web_portal", exp, iat}`.
Drop the `purpose` and `redirect` claims; the redirect target travels
as a URL query parameter (`?redirect=<encoded>`) — same as the
existing FE flow in `useOpenInWebPortal.ts:22`. This:
- keeps `consume-ticket` untouched (one-shot consume + audience
  pinning + 6-fail rate limit reuse "for free");
- avoids minting a second consume endpoint;
- keeps PII boundary identical to the existing ticket bridge — the
  ticket carries `tg` (telegram_id) and `plan` already, no new
  surface;
- requires the new endpoint to look up the user by `telegram_id`
  (DB hit) to populate `sub` (user.id) and `plan` — this is what
  the 404-on-unknown branch in A2 handles anyway.

### O.2 — Mini_app `OwnPayouts` "Запросить вывод" button breaks if placeholder is removed (medium)

**Issue.** Plan A6 says "remove `OwnPayoutRequest` entirely … никакого
'keep as fallback'." But `mini_app/src/screens/owner/OwnPayouts.tsx:83`
imperatively calls `navigate('/own/payouts/request')` from inside the
mini_app — there is no bot keyboard involvement. Removing the
placeholder turns that click into a 404 (NotFoundScreen).

**Recommendation (in-scope adjustment).** In Commit 3, replace the
imperative `navigate(…)` with the `useOpenInWebPortal('/own/payouts/request')`
hook (same hook the placeholder used internally). This keeps the
mini_app `OwnPayouts` list functional after placeholder removal,
without re-introducing the placeholder file. ~5 line change in
`OwnPayouts.tsx`. Pattern matches existing
`mini_app/src/screens/common/MainMenu.tsx` and
`mini_app/src/screens/common/Cabinet.tsx` where PII flows already
use `useOpenInWebPortal` directly from a list/menu screen.

### O.3 — Bot must reach API in-network, not via nginx (low / config hygiene)

**Issue.** Plan A5 says "HTTP POST to API endpoint." The natural
default (`settings.api_public_url`,
`https://api.rekharbor.ru`) goes through nginx + Cloudflare — adding
TLS handshake + nginx hop for what is structurally an intra-cluster
call. Worse, public nginx has its own rate-limit / WAF rules that
could throttle bot calls under load and make this path fragile.

**Recommendation (Type 2 ADAPT).** Add a dedicated Settings field
`internal_api_base_url: str = Field("http://api:8001", alias="INTERNAL_API_BASE_URL")`.
Bot uses this to call API directly via Docker DNS. The public
URL stays untouched as `api_public_url`. Container env is already
shared via `.env`, so this works in dev (override to
`http://localhost:8001`) and prod (default `http://api:8001`)
without code change.

### O.4 — `BOT_TOKEN` as HMAC key conflates two different secrets (medium)

**Issue.** A1 reuses `BOT_TOKEN` as the HMAC shared secret. `BOT_TOKEN`
is also (a) the Telegram-API authentication credential aiogram uses for
every Bot HTTP call, (b) the secret used to sign Telegram `init_data`
(see `auth_utils.validate_telegram_init_data` line 60 — it derives
`HMAC("WebAppData", BOT_TOKEN)` as that secret).

Reusing `BOT_TOKEN` as a *third* purpose (server-to-server HMAC)
means any future audit / rotation of `BOT_TOKEN` simultaneously
breaks (a)+(b)+the bot-to-portal exchange — a single rotation
event must be co-ordinated across three protocols.

The plan's rationale ("shared-secret separate from BOT_TOKEN =
ненужное дублирование") underweights this. A separate HMAC key has
the same operational cost (one extra `.env` field, one extra
container-side mount) but cleanly separates concerns.

**Verdict.** Marina's "no new secret env vars" constraint
(prompt line 207) overrides this. Recording as a known asymmetry —
ops doc should call out that rotating `BOT_TOKEN` requires
coordinated bot+API restart. **Defer** to a follow-up BL if the
team later wants to split secrets; not blocking BL-055.

### O.5 — Ruff baseline drift since prompt was written (low / housekeeping)

**Issue.** Prompt says ruff baseline = 20 errors. Verified locally
