## 3. Gate framework integration point

**Enums:**
- `PlacementGate(StrEnum)` в `src/core/enums/placement_gate.py:18` — **НЕ `GateCode`** (prompt L37 terminology устарела); G18 highest, **G19 свободен** ✅
- `GateReason(StrEnum)` в `src/core/enums/gate_reason.py:13` — snake_case i18n keys

**Structure** (`src/core/services/legal_compliance_service.py`):
- `_GATE_CHECKERS: dict[PlacementGate, (session, placement) → GateResult]` — placement-side
- `_USER_GATE_CHECKERS: dict[PlacementGate, (session, user) → GateResult]` — user-side (5b.7a addition)
- `_TRANSITION_GATES`: resolution table per `(from, to)` status pair
- `_USER_ROLE_GATES`: resolution table per `role` (`owner` | `advertiser`)

**Existing G04 pattern:** shared body `_check_gNN_for_user_id(session, user_id)` + two thin wrappers (`check_gNN(session, placement)` resolves `placement.owner_id`; `check_gNN_user(session, user)` accepts directly).

**`ChannelAddDeclinedError(ForbiddenError)`** в `src/core/exceptions.py:282`.

**⚠️ Architectural problem для G19** (см. §9 S5): existing `_USER_GATE_CHECKERS` signature не принимает channel context. G19 нужно `member_count` ДО channel creation. 3 options enumerated (a/b/c в tmp `bl107_03_gate_framework.md`). **Recommend (b)**: parallel `_CHANNEL_CONTEXT_GATE_CHECKERS` registry с `(session, user, channel_data) → GateResult` signature + new orchestration `check_gates_for_channel_add(user, channel_data) -> list[GateResult]`.

**New `GateReason` codes для G19:**
- `BLOGGER_REGISTRY_NOT_VERIFIED`
- `BLOGGER_REGISTRY_PENDING_REVIEW`
- `SUBSCRIBER_COUNT_UNKNOWN` (Telegram API failure during check)
- Threshold-not-applicable: reuse `OK` с `reason_code=OK, passed=True` для <10k channels (gate body short-circuits)
